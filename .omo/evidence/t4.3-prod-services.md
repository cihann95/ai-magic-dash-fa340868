## 1. Sentry (Error Monitoring)

### 1.1 Current State

| Item | Detail |
|------|--------|

| `.env` value | `https://dev@sentry.example.com/1` (placeholder) |
| Code integration | **NOT INTEGRATED** — `src/lib/observability.ts` uses `console.log/warn/error/debug` only |
| SDK dependency | `@sentry/*` packages are **NOT** in `package.json` |
| Frontend config | `src/lib/config.ts` does **NOT** reference `SENTRY_DSN` |

**⚠️ GAP:** `SENTRY_DSN` exists in `.env.example` but no code reads it. The observability module (`src/lib/observability.ts`) is a thin console logger — it does not initialize or send events to Sentry. Before production, Sentry SDK integration must be added.

### 1.2 Setup Procedure

1. **Create Sentry project**
   - Go to [sentry.io](https://sentry.io) → Create account / log in
   - Create a new project (suggested platform: `javascript` for frontend, `node` for edge functions)
   - Note the **DSN** from Settings → Projects → Client Keys (DSN)

2. **Install Sentry SDK**
   ```bash
   # Frontend
   npm install @sentry/react

   # Edge Functions (Deno)
   # Add to supabase/functions/_shared/sentry.ts:
   # import * as Sentry from "https://esm.sh/@sentry/browser@8";
   ```

3. **Initialize Sentry in app**
   ```typescript
   // src/lib/observability.ts — add Sentry init
   import * as Sentry from "@sentry/react";

   if (import.meta.env.VITE_SENTRY_DSN) {
     Sentry.init({
       dsn: import.meta.env.VITE_SENTRY_DSN,
       environment: import.meta.env.MODE,
       tracesSampleRate: 0.1,
     });
   }
   ```

4. **Set environment variable**
   - Frontend (Vite): `VITE_SENTRY_DSN=https://...@sentry.io/...`
   - Edge Functions: `SENTRY_DSN=https://...@sentry.io/...` (set via Supabase Dashboard → Secrets)

5. **Add to `.env.example`** (if not already present — currently only `SENTRY_DSN` exists, `VITE_SENTRY_DSN` is missing)

|---|------|-------|-----------------|

| 3 | **Performance trace** | Navigate through 3+ pages | Transaction traces appear in Sentry Performance tab |

## 2. VAPID Keys (Web Push Notifications)

### 2.1 Current State

| Item | Detail |
|------|--------|
| Env vars | `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT` |
| `.env` values | `local-dev-vapid-*` (placeholders) |
| Frontend | `src/pages/Settings.tsx` — `VAPID_PUBLIC_KEY = ""` **HARDCODED EMPTY** |
| Frontend | `src/lib/pushSubscribe.ts` — `enablePushNotifications(vapidPublicKey)` |
| Backend | `supabase/functions/send-push/index.ts` — reads `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT` from `Deno.env.get()` |
| Backend config | `supabase/functions/_shared/config.ts` — all three VAPID vars are **optional** |

**⚠️ GAP:** `Settings.tsx` line 17 has `const VAPID_PUBLIC_KEY = ""` — this is hardcoded to empty string and never reads from `import.meta.env`. Push notifications will always show "Admin must add VAPID key" until this is fixed. A `VITE_VAPID_PUBLIC_KEY` env var must be added and wired into the frontend.

### 2.2 Setup Procedure

1. **Generate VAPID keys**
   ```bash
   npx web-push generate-vapid-keys
   ```
   Output:
   ```
   Public Key:  BFxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
   Private Key: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
   ```

2. **Set environment variables**

   | Variable | Where | Value |
   |----------|-------|-------|
   | `VAPID_PUBLIC_KEY` | Supabase Edge Function Secrets | Generated public key |
   | `VAPID_PRIVATE_KEY` | Supabase Edge Function Secrets | Generated private key (**SECRET**) |
   | `VAPID_SUBJECT` | Supabase Edge Function Secrets | `mailto:admin@lumen.trade` |
   | `VITE_VAPID_PUBLIC_KEY` | Vite build env / hosting env | Same public key (browser-safe) |

3. **Fix frontend VAPID key injection** (code change required)
   ```typescript
   // src/pages/Settings.tsx — line 17
   // BEFORE:
   const VAPID_PUBLIC_KEY = "";
   // AFTER:
   const VAPID_PUBLIC_KEY = import.meta.env.VITE_VAPID_PUBLIC_KEY ?? "";
   ```

4. **Add to `.env.example`**
   ```
   VITE_VAPID_PUBLIC_KEY=your_vapid_public_key
   ```

5. **Set Supabase secrets**
   ```bash
   npx supabase secrets set \
     VAPID_PUBLIC_KEY="<generated-public-key>" \
     VAPID_PRIVATE_KEY="<generated-private-key>" \
     VAPID_SUBJECT="mailto:admin@lumen.trade" \
     --project-ref <project-ref>
   ```

## 3. Upstash Redis (Rate Limiting)

### 3.1 Current State

| Item | Detail |
|------|--------|
| Env vars | `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN` |
| `.env` values | `http://localhost:6379` / `local-dev-redis-token` (local-dev placeholders) |
| Backend | `supabase/functions/_shared/redis.ts` — Redis client with fail-open |
| Backend | `supabase/functions/_shared/rate-limit.ts` — sliding window rate limiter |
| Backend config | `supabase/functions/_shared/config.ts` — both vars are **optional** |
| Fail-open | When Redis is unavailable, `redisEnabled = false` → all rate limits pass through |

**Rate-limited endpoints (7 functions):**

| Function | Window | Max Requests | Key Prefix |
|----------|--------|-------------|-------------|
| `execute-trade` | 1 min | 20 | `rl:exec` |
| `blitz-tick-order` | 1 min | 30 | `rl:blitz-tick` |
| `blitz-matchmake` | 1 min | 10 | `rl:blitz-match` |
| `ai-analyze` | 1 min | 15 | `rl:ai` |
| `ai-strategy` | 1 min | 15 | `rl:ai` |
| `ai-chat` | 1 min | 15 | `rl:ai` |
| `ai-trade-coach` | 1 min | 15 | `rl:ai` |

Default (unlisted functions): 1 min / 30 requests / `rl:default`

### 3.2 Setup Procedure

1. **Create Upstash Redis database**
   - Go to [console.upstash.com](https://console.upstash.com)
   - Create a new Redis database (select region closest to Supabase edge function region)
   - Note the **REST API URL** and **REST API Token** from the dashboard

2. **Set environment variables**

   | Variable | Where | Value |
   |----------|-------|-------|
   | `UPSTASH_REDIS_REST_URL` | Supabase Edge Function Secrets | `https://xxx.upstash.io` |
   | `UPSTASH_REDIS_REST_TOKEN` | Supabase Edge Function Secrets | Upstash REST token (**SECRET**) |

3. **Set Supabase secrets**
   ```bash
   npx supabase secrets set \
     UPSTASH_REDIS_REST_URL="https://xxx.upstash.io" \
     UPSTASH_REDIS_REST_TOKEN="<your-token>" \
     --project-ref <project-ref>
   ```

4. **Verify connectivity**
   ```bash
   # Test REST API directly
   curl -H "Authorization: Bearer <token>" \
     "https://xxx.upstash.io/ping"
   # Expected: {"result":"PONG"}
   ```

### 3.3 Production Validation

| # | Test | Steps | Expected Result |
|---|------|-------|-----------------|
| 1 | **Rate limit trigger** | Send 16+ rapid requests to `ai-analyze` (limit: 15/min) | 16th request returns `429` with `Retry-After` header |
| 2 | **Rate limit headers** | Send valid request to rate-limited endpoint | Response includes `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` headers |
| 3 | **Fail-open** | Temporarily set invalid Redis URL → send request | Request succeeds (rate limit bypassed gracefully) |
| 4 | **Redis connectivity** | Check Upstash dashboard → CLI | Keys matching `rl:*` pattern appear with correct TTLs |

### 3.4 Fail-Open Behavior

When Redis is unavailable (`redisEnabled = false`), the system degrades gracefully:

```typescript
// supabase/functions/_shared/redis.ts
export const redisEnabled = !!(URL && TOKEN);
// When false: all redis operations return fallback values
// rate-limit.ts: returns { allowed: true, ... } when redis disabled
```

This means:
- **No Redis configured** → all requests pass (no rate limiting)
- **Redis temporarily down** → all requests pass (fail-open)
- **Redis configured and up** → rate limiting enforced

- `supabase/functions/_shared/redis.ts` — Redis client wrapper (fail-open pattern)
- `supabase/functions/_shared/rate-limit.ts` — sliding window rate limiter
- `supabase/functions/_shared/config.ts` — config (optional vars)

## 6. Security Notes

- **NEVER** commit `.env` to version control (already gitignored)
- **NEVER** expose `VAPID_PRIVATE_KEY` to the browser — it is server-side only
- **NEVER** share `UPSTASH_REDIS_REST_TOKEN` publicly
- **ROTATE** VAPID keys periodically (quarterly recommended)
- **RESTRICT** Sentry DSN to known origins via Content Security Policy
- **USE** separate Upstash databases for staging vs production
- **USE** separate Sentry projects for staging vs production
- **MONITOR** Upstash usage — rate limiting costs scale with request volume
- **BACKUP** VAPID keys securely — regenerating them invalidates all existing push subscriptions
